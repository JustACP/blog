数据库基础

数据库是什么

数据库是数据管理的有效技术，是由一批数据构成的有序集合，数据存放在数据表里。数据表相互关联，反应客观事物之间的本质联系。

在JavaSE学习之中，我们学习了如何使用文件I/O 将数据保存到本地，这样就可以将数据持久地储存在本地，即使程序重新打开也可以加载上一次数据。当我们数据非常多的时候，这个方法就不太方便了。查找其中的某一个数据，就只能加载到内存查找，很难受的。

数据库就可以快速的增删查改。

常见的数据库

常见的数据有多种包括但是不限于：

  • MySQL- 免费、使用惯犯、开源数据库、中小型
  • MS SQL - 收费、提供技术支持、适用Windows Server
  • Oracle -收费、技术支持、大型数据库

我们学习的是MySQL ,SQL语句大部分是通用的。(我安装的 Mariadb，一个开源系统)

数据模型

数据模型,和现实世界一样.对现实世界特征的一种抽象. 有以下特点:

  • 属性不可再分
  • 一个实体属性有多个
  • 用于唯一区分不同实体属性, 称之为Key,例如身份证号
  • 属性取值可以有一定约束, 比如性别

实体或是属性之间可以有一定联系,比如一个老师教很多学生, 而学生相对于老师就是被教授的关系.每个学生有一个唯一学号与其对应.(1:1).

MySQL 是一种关系型数据库,通过使用关系型数据库,就可以很好的储存这些带有一定联系的数据.

数据库的规范化

我们要设计一个实际表，我们就要尽量减少不好的关系。

第一范式（1NF）

第一范式是指数据库每一列都是不可分割基本数据项。下面这个就存在可分割情况：

  • 学生（姓名、电话号码）
    电话号码实际包括了 家用座机电话和 移动电话，是可拆分的。
  • 学生（姓名、座机号码、手机号码）

第二范式(2NF)

要求表里必须存在主键，其他属性必须完全依赖主键：

  • 学生（<u>学号</u>, 姓名，性别）
    学号就是学生的唯一标识，每个学生有不同学号，因此表中存在一个主键。学号改变其他信息随之改变。

第三范式（3NF）

满足第二范式的情况下，所有属性都不传递依赖主键，满足第三范式：

  • 学生借阅情况（<u>借阅编号</u>，学号，书籍编号，书籍名陈，书籍作者）
    实际上书籍编号依赖于借阅编号，而书籍名称和书籍作者依赖于书籍编号，因此存在传递依赖的情况，我们可以单独拆分成另一张表：
  • 学生借阅情况（<u>借阅编号</u>，学号，书籍编号）
  • 书籍（<u>书籍编号</u>，书籍名陈，书籍作者）

（管理员ID,储存物品ID) -> （仓库ID,数量)

因此（仓库ID,物品ID）和 （管理员ID,储存物品ID) 都是StorehouseManage的候选关键字，表中唯一的非关键字段为数量，符合第三范式。但是存在如下决定关系：
（仓库ID）-> (管理员ID)
(管理员ID) -> (仓库ID)
即使存在关键字段决定关键字段的情况，若修改管理员id，就必须逐一进行修改，所以不符合BCNF范式.
